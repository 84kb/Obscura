## 3. 実行順序と依存関係
1. `DEV-A1/A2/A4` と `DES-P1/P2` を先行完了（R1-R4の判定材料を早期確保）。
2. `DEV-A3` と `DES-P3` でR5の置換仕様を確定。
3. `QA-S1~S4` は上記差分を基準にE2E/回帰を実施。
4. `OPS-T2/T3` で移行・監視・ロールバックを確定し、`PLN-G2` で最終再提出パッケージ化。

## 4. リリースゲート（全項目必須）
- Gate-1: 右クリックメニュー表示ログ `0` 件（Windows/macOS）。
- Gate-2: `dominant_color` 一致率 `100%`（20サンプル）。
- Gate-3: `.library` 表示 `0` 件（一覧/詳細/検索）。
- Gate-4: 孤立UUIDが起動時に自動除外され、自己修復ログが残る。
- Gate-5: OSネイティブ追加ウィンドウ/ダイアログ呼び出し `0` 件。
- Gate-6: QA正常/異常ケースが各 `10` 件以上で再発 `0` 件。
- Gate-7: 移行手順書とロールバック手順が運営承認済み。

## 5. 再提出パッケージ要件（最終承認前提）
- 開発: 実装差分一覧、API契約差分、実行ログ。
- デザイン: UI仕様差分、画面別適用マトリクス。
- QA: E2E/回帰結果表、失敗時再現手順、再試験結果。
- 運営: 移行手順書、監視ダッシュボード項目、ロールバック条件。
- 企画: R1-R5の充足判定シートと承認判断メモ。

## 6. 判定ルール（本ラウンド）
- CRITICAL/HIGH: 即時是正対象（本ドキュメントのR1-R5は全て該当）。
- MEDIUM/LOW: 警告管理のみ（次ラウンド計画へ送る。実装修正は行わない）。
